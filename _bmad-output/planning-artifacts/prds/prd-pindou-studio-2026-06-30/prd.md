## 1. Vision

拼豆爱好者最高的一道坎，不在"图转得准不准"，而在"设计完了，手头的料够不够拼"。今天的工具默认"先有图、再买料"；当用户发现缺某个色，市面上的竞品（BeadMaster、Fuse Beads Studio 等）的回答都是同一句——**"去补货"**。

> **诚实声明（评审要求）**：上述痛点是基于头脑风暴与领域直觉的**合理假设**，`[ASSUMPTION]` **尚未经用户访谈或社区数据证实**（市场研究阶段 Reddit/小红书等 UGC 渠道抓取受限）。"竞品只做补货、无人做感知替代"亦**仅据 App Store 截图初步确认，待动工前人工深核**（见 Open Question 4/5）。本 PRD 据此假设推进；但这些假设是整个方向的根基，须在投入前或早期用最小成本验证。

库存驱动设计反转这个默认：**先告诉工具你手头有哪些珠、各多少，工具在这套真实库存的约束下帮你把图做出来**。当某个目标色你没有或不够时，pindou-studio 不让你停工等快递，而是用 CIEDE2000 感知色差，从**你已经拥有的颜色里**挑出视觉损失最小的替代色——**别人说"去买"，我们说"用手头的替"**。

这是 pindou-studio 已有的 CIEDE2000 引擎"反着用"的第一次落地：把颜色匹配的色板从"理想全色板"换成"用户库存子集"，缺色时做感知最优替代。它复用现有引擎、保持纯前端零安装的形态，并在竞品集体停留于"库存盘点 + 补货清单"的当下，切入一个（据现有核实）无人涉足的深度——**科学的缺色替代**。

**差异化窗口是窄的、且脆弱**：库存驱动本身不是空白（竞品已做得很深），pindou 的差异化全押在"感知替代（替代而非补货）"+"Web 端零安装、隐私本地化"（竞品均为国内移动 App，此形态不易顺手复制）这两点深度上。这层窗口竞品随时可能跟进——立论靠的是"做得更深、做得更早"，不是"独此一家"。跨品类已有先例佐证此范式有真实刚需：乐高 **Rebrickable** 围绕"用你拥有的零件能拼什么"规模化运营多年。

## 2. Target User

### 2.1 Jobs To Be Done

- **功能性**：把一张想做的图，在"我手头实际有的珠子"约束下，转成一份**真的能开工**的图纸——不缺料、或缺了也知道怎么用现有料替代。
- **功能性**：搞清楚"我现在的库存到底能不能拼这张图"，缺哪几色、缺多少，一目了然。
- **情感性**：消除"兴冲冲设计完、一摆才发现没料"的扫兴；减少"肉眼硬凑替代色结果偏色难看"的挫败。
- **情感性（省）**：不为单张图重复补货、不让买回来的珠子在抽屉里吃灰。
- **builder 自身**：维护者本人也是拼豆玩家，这是"为自己做的工具"，自用体验是第一验收人。`[ASSUMPTION]`

### 2.2 Non-Users (v1)

- **手作卖家 / 接单做图纸的商家**：他们要的是批量、对账、采购流，不是个人库存创作闭环——v1 不为他们优化。
- **纯新手 / 儿童**：降门槛是「极简风格化」方向（未来）的事，本方向假设用户已有一定库存与经验。
- **不记库存的随性玩家**：录入库存对他们是纯负担，本方向对他们价值有限——可继续用基线转珠功能，不强推库存。

### 2.3 Key User Journeys

- **UJ-1. 林夕想用抽屉里现有的珠子做一张初音图，而不是又下一单。**
  - **Persona + context**：林夕，工作党拼豆玩家，攒了二十来个色号、几千颗珠，抽屉里有存货但懒得为每张新图补货。
  - **Entry state**：打开 pindou-studio（浏览器，无需登录），已在之前某次录好库存；进入编辑器，导入一张初音的图。
  - **Path**：① 转图后，开启「库存模式」；② 工具用她的库存色板重新匹配，画布上立刻标出**缺色格子**（高亮）和**库存不足的色**（红字）；③ 她点"用库存替代"，工具把每个缺色格用 CIEDE2000 在她拥有的色里找到的最近色替换，并给出"替代了 N 格、视觉影响预览"。
  - **Climax**：材料清单从"缺 3 色、需补货"变成"全部可用你现有库存完成"——她不用等快递，今晚就能开摆。
  - **Resolution**：导出带色号的 PDF（沿用现有导出），按图施工。**Edge case**：某个色缺口太大、连替代都不够（库存数量不足），工具明确标"此色即便替代后仍差约 X 颗"，让她决定是补货还是改图。

- **UJ-2. 林夕在做之前先问一句"这张图我现在能拼吗"。**
  - **Persona + context**：同上，她从社区存了一张图纸，想知道不补货能不能直接开工。
  - **Path**：导入图纸 → 开库存模式 → 工具给出**可拼性判定**：可拼 / 缺色可替代后可拼 / 数量不足不可拼，并列出缺口清单（缺哪些色、各差多少）。
  - **Climax + Resolution**：一眼看到"绿灯：可替代后可拼"，放心开工；或"黄灯：差 H2 约 1200 颗"，决定补货还是换图。

## 3. Glossary

*下游工作流与读者须逐字使用这些术语。FR / UJ / SM 一律用术语表词汇，全文不得引入同义词。*

- **库存（Inventory）** — 用户声明的、自己实际拥有的珠子集合。由若干**库存条目**组成。绑定到某个**品牌色板**。
- **库存条目（Inventory Entry）** — 一条记录：`{色号, 拥有数量}`。色号取自某个品牌色板；拥有数量为非负整数（0 视为"曾登记但已用完"，与"从未登记"区分）。
- **品牌色板（Brand Palette）** — 现有概念（Perler / Hama / Artkal 等），一组带色号与 hex 的颜色全集。库存是品牌色板的子集（带数量）。
- **库存色板（Inventory Palette）** — 由库存推导出的"约束色板"：品牌色板中用户拥有数量 > 0 的颜色子集。库存驱动匹配的实际作用色板。
- **库存模式（Inventory Mode）** — 编辑器的一个开关。开启后，颜色匹配、可拼性、材料清单都以**库存色板**而非完整品牌色板为基准。
- **缺色（Missing Color）** — 当前图纸用到、但不在库存色板里（拥有数量为 0 或从未登记）的颜色。
- **数量不足（Insufficient）** — 在库存色板里、但图纸所需用量 > 拥有数量的颜色。
- **感知替代（Perceptual Substitution）** — 对一个缺色，用 CIEDE2000 从**库存色板**中选出感知色差最小的颜色来替换其全部格子。区别于竞品的"补货建议"。
- **可拼性判定（Buildability Verdict）** — 对"当前图纸 × 当前库存"的三态结论：**可拼** / **替代后可拼** / **数量不足不可拼**，附缺口清单。
- **缺口清单（Shortfall List）** — 缺色与数量不足的明细：每项 `{色号, 类型(缺色/不足), 差额数量}`。

## 4. Features

### 4.1 库存管理（录入与维护）

**Description:** 用户以最低成本把"我有哪些珠、各多少"告诉工具。录入方式为**按品牌色板勾选 + 填数量**：选定一个品牌色板后，工具列出该色板全部色号，用户勾选自己拥有的、为每个勾选项填一个大概数量。库存按品牌色板独立保存，本地持久化，可随时回来增删改。实现 UJ-1、UJ-2 的前置状态。使用术语表中「库存」「库存条目」「品牌色板」。

**Functional Requirements:**

#### FR-1: 按色卡勾选录入库存

用户可在选定**品牌色板**后，从其全部色号中勾选拥有的颜色，并为每个勾选的色号填写**拥有数量**（非负整数）。Realizes UJ-1, UJ-2.

**Consequences (testable):**
- 选定某品牌色板后，界面列出该色板的全部色号，每项带色块、色号与可选名称。
- 勾选某色号后可输入数量；未勾选的色号不计入库存。
- 数量字段接受 0 与正整数；输入非整数或负数时予以拒绝或归一化，并给出明确反馈。
- 录入完成后，系统生成对应的**库存条目**集合，构成该品牌色板下的**库存**。

#### FR-2: 库存本地持久化与维护

用户的库存在**本地持久化**，跨会话保留，并可随时增删改。Realizes UJ-1.

**Consequences (testable):**
- 关闭并重开浏览器后，已录入的库存仍在；本地存储，无需登录。
- 用户可修改任一**库存条目**的数量、取消勾选以移除条目，或新增勾选。
- 同一用户可分别保存多个**品牌色板**各自的库存，互不覆盖。
- `[ASSUMPTION]` 清空或重置库存须二次确认，避免误删。

#### FR-3: 库存数量的快捷批量操作

用户可对库存做轻量批量操作以降低录入成本。Realizes UJ-1.

**Consequences (testable):**
- 支持"全选本色板 / 全不选"。
- 支持给一批勾选项设同一默认数量（如"我每色大约 1000 颗"），再个别微调。

**Notes:** `[NOTE FOR PM]` 录入成本是本方向最大的体验风险（简报已标）。FR-1/3 的目标是让"录入一板新货"在一两分钟内完成；若 UX 测试显示仍嫌繁重，可考虑日后用拍照或订单截图辅助——属未来增强，非 MVP，且与「实拍」方向相关。

### 4.2 库存模式与库存约束匹配

**Description:** 编辑器新增「库存模式」开关。开启后，颜色匹配、画布渲染、材料清单全部以**库存色板**（库存中数量 > 0 的颜色子集）为基准，而非完整品牌色板。这是把现有 CIEDE2000 引擎"换输入"的核心落地——色板从理想全集换成用户真实拥有的子集。实现 UJ-1、UJ-2。须遵守 project-context 的三不变量（grid 行列、countMap、颜色索引范围），且切换库存模式应经 store 的合法路径（如 `commitConversion`），不绕过 `applyCellChanges`。

**Functional Requirements:**

#### FR-4: 库存模式开关

用户可在编辑器中开启/关闭**库存模式**，针对当前选定的**品牌色板**与其**库存**。Realizes UJ-1, UJ-2.

**Consequences (testable):**
- 库存模式默认关闭；关闭时，行为与现有基线转珠完全一致，无任何回归。
- 开启库存模式要求当前品牌色板已有非空库存；库存为空时，引导用户去录入（FR-1）。
- 开关状态在当前设计会话内保持不变。

#### FR-5: 以库存色板重新匹配图纸

开启库存模式时，用户可让工具用**库存色板**对当前图纸重新做颜色匹配。Realizes UJ-1.

**Consequences (testable):**
- 重新匹配仅在**库存色板**（拥有数量 > 0 的颜色）范围内用 CIEDE2000 选色。
- 重新匹配是显式动作，写回 grid 须经 store 合法路径提交（`commitConversion`），保持三不变量。
- 重新匹配可撤销，沿用现有 undo/redo，记为单条历史。
- **Out of Scope:** 切换品牌色板时不自动重匹配，沿用现有"切 palette 不重匹配"的既定行为，避免意外破坏。

#### FR-6: 缺色与数量不足的可视标记

开启库存模式时，工具在画布与材料清单上标出**缺色**与**数量不足**的颜色。Realizes UJ-2.

**Consequences (testable):**
- 画布上**缺色**格子带可辨识的视觉标记（如高亮或描边），与正常格区分。
- 材料清单中，**数量不足**的色号显示"需求 vs 拥有"并标红；**缺色**单独归类。
- 标记实时反映当前 grid 的真实用量（依赖 countMap 不变量），编辑后即时更新。
- `[ASSUMPTION]` 色盲友好：标记不只靠颜色，另辅以图标或纹理。

### 4.3 感知替代（差异化命脉）

**Description:** 这是 pindou-studio 区别于所有竞品的核心。对每个**缺色**，工具用 CIEDE2000 从**库存色板**中找到感知色差最小的颜色作为**感知替代**，并把该缺色的全部格子替换为替代色。用户能预览影响、确认或撤销、并清楚看到"哪些格被替代、视觉影响多大"。竞品对缺色只会"建议补货"；这里给"用手头色替代"。实现 UJ-1。使用术语表「感知替代」「缺色」「库存色板」。

**Functional Requirements:**

#### FR-7: 缺色的感知替代建议

对当前图纸的每个**缺色**，工具用 CIEDE2000 在**库存色板**中计算并给出感知色差最小的替代色。Realizes UJ-1.

**Consequences (testable):**
- 对每个缺色，输出一个来自库存色板的替代色号及其 CIEDE2000 色差值（ΔE）。
- 替代候选只取**库存色板**内的颜色，保证替代后确有料可用。
- 库存色板为空（无任何可用色）时，不产出替代，并提示用户去录入库存。
- 计算复用现有引擎：先用 `paletteLabs()` 预计算库存色板 Lab，再用 `findClosestColor`；遵守 project-context 性能约定，无需 Web Worker。

#### FR-8: 应用感知替代到图纸

用户可一键把所有（或选定的）**缺色**按 FR-7 的建议替换为库存色，写回图纸。Realizes UJ-1.

**Consequences (testable):**
- 应用替代后，被替代格子的颜色索引指向库存色板中的替代色；写回经 store 合法路径，保持三不变量。
- 应用记为单条撤销历史，可整体回退到替代前。
- 应用后，材料清单与可拼性判定（FR-10）即时刷新。
- **Out of Scope:** v1 不做"逐色手动指定替代色"的精细编辑，仅做基于 CIEDE2000 的自动建议与应用（用户仍可事后用现有画笔或取色手动改）。`[NON-GOAL for MVP]`

#### FR-9: 替代影响预览

用户在应用感知替代前，可预览替代的影响范围与程度。Realizes UJ-1.

**Consequences (testable):**
- 预览显示：将被替代的缺色数、受影响格子总数，以及每个替代的 ΔE 与"视觉影响：低/中/高"分级。
- **ΔE 分级初值（可推翻，动工前标定——见 Open Question 3）**：基于 CIEDE2000 经验区间给出起点——**低 ΔE < 2（肉眼几乎无差）/ 中 2–5（仔细看可辨）/ 高 > 5（明显不同色）**。此初值须用几张真实图纸标定后定稿，并作为 SM-C1 反指标的判据来源。`[ASSUMPTION]`
- `[ASSUMPTION]` 提供"替代前 / 替代后"画布对比（切换或并排），供用户判断是否接受。
- 用户可接受全部替代，或放弃（不改图纸）。

### 4.4 可拼性判定

**Description:** 把"我现在能不能拼这张图"做成一眼可读的三态结论 + 缺口清单。这是 UJ-2 的核心，也是与竞品对齐的能力（竞品已有"图纸可拼状态"），但这里把它和感知替代联动：判定会区分"缺色但可替代后可拼"与"数量真的不够"。实现 UJ-2。使用术语表「可拼性判定」「缺口清单」。

**Functional Requirements:**

#### FR-10: 三态可拼性判定

开启库存模式时，工具对"当前图纸 × 当前库存"给出**可拼性判定**：可拼 / 替代后无缺色 / 数量不足。Realizes UJ-2.

**Consequences (testable):**
- **可拼（绿）**：图纸所有颜色都在库存色板内，且每色拥有量 ≥ 需求量。
- **替代后无缺色（黄）**：存在缺色，但全部缺色都能在库存色板找到替代——即替代后图纸用到的每个色都在你的库存里。**承诺边界**：黄灯只保证"没有你缺的色了"，**不保证数量精算到颗够完工**（替代会把缺色格子并入替代色，可能令替代色用量逼近或略超其拥有量）。数量层面的余量由 FR-11 缺口清单如实呈现。
- **数量不足（红）**：图纸存在颜色既不在库存色板内、也无任何可替代色（库存色板放不下它）。
- 判定结果以红/黄/绿状态清晰呈现，并配一句话解释。
- **设计取舍（party-mode 决议）**：v1 **不做替代色库存的级联校验**——不承诺"精算到颗够完工"。理由是产品定位而非技术限制：差异化命脉是"用手头色替代、不停工"，不是"保证你库存够铺满整图到颗"（那是竞品补货助手的领域，已列 Non-Goal）。数量透明交由 FR-11。

#### FR-11: 缺口清单

可拼性判定附带**缺口清单**，列出每个问题色的明细。Realizes UJ-2.

**Consequences (testable):**
- 每项含：色号、类型（缺色 / 数量不足）、差额数量（不足多少颗 / 缺色需多少颗）。
- **每行必须带人类可读的解读，不能只甩数字（party-mode 决议）**：在"需求 2000 / 拥有 1900"这类数字旁，缀一句引擎算得出、用户读得懂的话——如「**够铺这张图，余量紧张**」或「**还差约 100 颗**」。理由：用户肉眼无法从"2000/1900"判断后果，引擎算得出的事不该甩回给用户用肉眼算；这是「信息透明」与「把账本甩脸上」的分界线。
- 缺色项可直接跳转到 FR-7 的替代建议。
- 数量不足项的解读句给出"还差约 X 颗"，供用户决定补货或改图。
- `[ASSUMPTION]` 缺口清单可复制或导出为文本，方便用户自行采购，但**不内置购买/采购链接**——与竞品"一键导出采购"不同，我们不做电商导流，保持工具中立轻量。

### 4.5 Cross-Cutting Constraints（现有架构铁律，下游必须遵守）

本方向引入大量新 UI 文案与新交互，须遵守 `project-context.md` 已确立的硬约束；违反将静默出错或阻断构建。

- **i18n（构建门禁）**：所有新增 UI 文案（库存模式开关、缺色/不足标记、三态灯、缺口清单人话解读、二次确认等）的每个文案 key **必须同时补 `en` 和 `zh`**——`zh` 类型由 `typeof en` 约束，漏一个即类型不过、阻断构建。非组件上下文用导出的 `t()`。
- **导出形态**：FR-11 缺口清单导出**仅限纯文本 / 剪贴板**，**不内嵌进 PDF**。若未来要进 PDF，中文必须经 `addLabel()` 渲染成 canvas 图——jsPDF 内置字体不支持 CJK，直接 `doc.text()` 会乱码。
- **性能**：库存色板重匹配与感知替代沿用现有路径——先用 `paletteLabs()` 预计算 Lab，再用 `findClosestColor`——作用于板尺寸级（数千格），**无需引入 Web Worker**（project-context 明确约定）。
- **状态变更合法路径**：所有写回 grid 的操作（重匹配 FR-5、应用替代 FR-8）须经 `commitConversion` / `applyCellChanges`，不绕过 store 直改 grid，以保持三不变量。

## 5. Non-Goals (Explicit)

- **不做补货/采购电商闭环**：不内置购买链接、不做"一键下单"。差异化是"用手头替代"，不是"帮你买"。竞品做导购，我们刻意不做。
- **不做账号体系与云端库存（v1）**：库存仅本地持久化。云同步是架构预留，非 v1 交付。
- **不做实拍识别库存**：拍照识别珠子属「实拍数字化」方向（未来、高风险），本 PRD 不含。
- **不做社区/图纸交易/库存分享**：避开生态层（简报战略）。
- **不改动基线转珠的默认行为**：库存模式是叠加的可选层，关掉后一切如旧。
- **不做多用户/协作库存**。

## 6. MVP Scope

### 6.1 In Scope

- 按色卡勾选 + 填数量的**库存录入**与本地持久化（FR-1, FR-2, FR-3）。
- **库存模式**开关 + 以库存色板重匹配（FR-4, FR-5）。
- **缺色 / 数量不足**可视标记（FR-6）。
- **感知替代**：建议 + 预览 + 应用（FR-7, FR-8, FR-9）—— 差异化命脉，必须做透。
- **可拼性判定** + 缺口清单（FR-10, FR-11）。
- 复用现有 PNG/PDF 导出（无新增，库存模式下导出的图纸即替代后的结果）。

### 6.2 Out of Scope for MVP

- 云同步 / 账号（架构预留，不实现）。`[NOTE FOR PM]` 录入数据本地存储有丢失风险（清缓存即丢），若用户反馈强烈，优先做"导出/导入 JSON 备份"作为轻量过渡，再考虑云同步。
- 逐色手动指定替代色的精细编辑（FR-8 Out of Scope）。
- 拍照/订单截图辅助录入（与实拍方向相关，未来）。
- 采购/补货导出链接、电商导流（明确 Non-Goal）。
- 多品牌色板混合库存的跨板替代（v1 假设替代只在同一品牌色板的库存色板内）。`[ASSUMPTION]`

## 7. Success Metrics

*pindou-studio 是个人开源项目、暂不变现，更看重使用信号与差异化是否被感知，而非营收。*

**Primary**
- **SM-1（差异化命脉被使用）**：库存模式会话中，**触发并应用过感知替代**的比例。目标：过半库存模式会话用到替代，说明"替代而非补货"确实解决了痛点。Validates FR-7, FR-8.
- **SM-2（闭环达成）**：开启库存模式的图纸中，最终达到"可拼 / 替代后可拼"绿/黄灯并导出的比例。Validates FR-10, FR-5, FR-8.

**Secondary**
- **SM-3（录入不劝退）**：开始录入库存的用户中，完成至少一个可用库存（≥1 条目且能开库存模式）的比例。Validates FR-1, FR-3.
- **SM-4（自用验收）**：维护者本人是否在真实拼豆中持续使用此功能（"我自己每次做图都开库存模式"）。Validates 整体。`[ASSUMPTION]`

**Counter-metrics (do not optimize)**
- **SM-C1（替代别糊弄）**：感知替代的平均 ΔE 不应为了"全绿"而无上限放大——"高视觉影响"（ΔE > 5，初值见 FR-9）的替代占比要低，防止为求"可拼"而给出难看的替代。Counterbalances SM-1。
- **SM-C2（别为库存牺牲基线）**：库存模式不得拖慢或劣化基线转珠体验；关闭库存模式时，性能与结果零回归。Counterbalances SM-2。

1. ~~**替代粒度**~~ **【已定 — 整色替换】**：缺色替代为"整色替换"（该缺色所有格子换成同一替代色），简单可预期。逐格/按区域不同替代不在 v1（用户仍可事后手动画笔微调）。

4. **【动工前必做 — 来自市场研究 🔴】竞品深核**：人工到 App Store 实际试用 BeadMaster / Fuse Beads Studio 等，确认它们的缺色处理确为"补货"、确实**未做 CIEDE2000 级感知替代**。这是整个差异化立论的命门——若已有竞品做了，需重估方向。（目前仅据商品页截图初步确认。）
