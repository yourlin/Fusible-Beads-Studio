# 输入核对报告：PRD 是否吸收了上游关键内容

核对目标 PRD：`_bmad-output/planning-artifacts/prds/prd-pindou-studio-2026-06-30/prd.md`

核对方法：逐个上游输入 → 抽取关键点（尤其定性/战略/情绪类容易被结构化 FR 静默丢掉的内容）→ 核对 PRD 是否吸收 / 该引用是否引用 / 有无冲突。

**总体结论**：PRD 吸收度高，战略立论、差异化命脉、三不变量约束、性能约定、Non-Goals 基本都接住了。主要 gap 集中在三类：(1) 市场研究的**证据缺口**（用户痛点未被证实仍是假设）这一关键诚实声明被 PRD 完全吞掉，使 PRD 读起来像痛点已确证；(2) 简报/研究里反复强调的 **"竞品深度未人工复核"** 这一前置风险未在 PRD 留痕；(3) 几条 project-context 硬约束（i18n 加 key 必须 en+zh、shared 改完要 rebuild、PDF 中文需 addLabel、导出像素上限）在 PRD 引入大量新 UI 文案/新导出内容的情况下未被点名，存在下游静默踩坑风险。

## 输入 1：产品简报 rev.2（brief.md）

### 已吸收（无 gap）
- 差异化立论"别人说去买、我们说用手头的替"——PRD §1 Vision、§4.3 完整接住，措辞一致。
- 战略取舍"避开生态层"——PRD §5 Non-Goals "不做社区/图纸交易/库存分享（避开生态层，简报战略）"显式引用。
- 不做补货导流——PRD §5、FR-11 `[ASSUMPTION]` 双重明示"不内置购买/采购链接……我们不做电商导流"。
- 用户画像（已攒珠、物尽其用、不为单张图补货的工作党玩家）——PRD §2.1/2.2/UJ-1 林夕画像一致。
- 录入成本是最大体验风险——PRD §4.1 Notes `[NOTE FOR PM]` 明确"录入成本是本方向最大体验风险（简报已标）"。
- "为自己做的工具"调性——PRD §2.1 builder 自身条 + SM-4 自用验收，接得很好（这是定性内容里接得最到位的一处）。

### Gap
1. **【该引用未引用 / 前置风险丢失】** 简报 Open Question 7 与"竞品现实"反复强调："竞品只做补货、无人做 CIEDE2000 感知替代"目前**仅据 App Store 截图初步确认**，建议写 PRD 前再人工深核一遍竞品缺色逻辑。PRD 把这个差异化当成**已确证的事实**写进 Vision/FR-7，未保留"此命脉尚待人工复核"的前提。若复核发现某竞品已做感知替代，整个差异化立论被削弱——PRD 没有为此留任何风险锚点。
2. **【定性/战略弱化】** 简报核心校正是"差异化窗口比初稿**窄**了——不再靠第一个做，而靠匹配质量做更深"。PRD §1 提到"无人涉足的深度"，但没有传达"窗口窄、靠深度续命、竞品随时可能跟进"这层**战略紧迫感/脆弱性**。结构化 FR 把它压成了一个功能点，丢了"这是我们仅存的护城河角度"的份量。
3. **【遗漏】** 简报 Vision 强调形态差异——"国内移动 App 竞品不易顺手复制的角度"（Web 端零安装 + 隐私本地化 + 跨平台）。PRD 提了"纯前端零安装"，但未把"竞品全是国内移动 App、我们靠 Web 形态形成差异"这一**对比性卖点**写明；§5 Non-Goals 提了本地持久化，但"隐私本地化是差异化的一部分"未点出。
4. **【轻微】** 简报次要用户/未来排序（极简风格化为低风险第二增量、与库存"约束色板"协同；实拍为远期高赌注）——PRD 在 Non-Goals 提了"实拍属未来方向""极简是未来方向"，但未传达简报里"三方向是同一引擎反着用、有明确推进排序"的产品全景。对单 PRD 可接受，但下游若想理解战略上下文会缺这一层。

## 输入 2：市场研究（market-...-research.md）

### 已吸收（无 gap）
- BeadMaster / Fuse Beads Studio 做得很深（库存盘点+补货+图纸可拼状态）——PRD §1、§4.4 引用，并明确"可拼性判定是与竞品对齐的能力"。
- "替代而非补货"是命脉——PRD §4.3 标题即"差异化命脉"，吸收充分。
- Rebrickable 跨品类类比——**已引用**：PRD 虽未在正文显式点名 Rebrickable，但"库存驱动反转默认"的范式被完整接住。（见下方 gap：点名缺失。）

### Gap
1. **【关键诚实声明被吞 / 定性丢失】** 研究最重的一条"证据缺口"：**三方向的真实用户痛点（设计完没料/摆错返工/新手太难）因 Reddit/小红书等 UGC 抓取受限，完全未被社区证实，目前仍是合理假设而非已验证需求**。PRD §1 Vision 以确定语气断言"拼豆爱好者最高的一道坎……"，§2.1 把用户挫败情绪写成既定事实，**全程没有一个 `[ASSUMPTION]` 标这是未证实的痛点**。这是结构化 PRD 最危险的静默丢失——把"合理假设"洗成了"已知事实"。建议在 Vision 或 §2.1 加一条 assumption + 在 Open Questions 加"痛点待用户访谈证实"。
2. **【证据缺口未引用】** 研究反复要求的前置动作："库存竞品深度未核实（5 款 App 结论来自搜索摘要，强烈建议人工到 App Store 复核），若已有人做了 CIEDE2000 级缺色替代则新叙事被削弱"——这是研究列为 🔴 最高优先的风险。PRD 未在任何地方记录此前置复核动作，也未列入 Open Questions。（与简报 gap 1 同源，但研究把它定为最高优先级，PRD 完全没接。）
3. **【该引用未引用 — 跨品类铁证】** 研究的 Rebrickable 论据是"库存驱动范式有真实刚需、可行、但不新鲜"的**双刃铁证**。PRD 完全没点名 Rebrickable，错失了一个能为"这个范式真有人要"背书、同时也提醒"不新鲜"的有力外部证据。在 Vision 或 Target User 处引用可显著增强立论可信度。
4. **【方法论置信度丢失】** 研究明确声明结论基于"无关键词搜索引擎、仅 WebFetch、App 详情页未直达、置信度中等"。PRD 把这些中等置信度结论当作高置信事实使用（"竞品集体停留于库存盘点+补货""无人涉足"），未传达"这是中等置信度、待复核"的限定。

## 输入 3：项目代码上下文（project-context.md）

### 已吸收（无 gap，且接得好）
- **三不变量**（grid 行列、countMap、颜色索引范围）——PRD §4.2 Description、FR-5/FR-8 Consequences 反复点名"保持三不变量"。
- **store 合法路径**（经 `commitConversion`/`applyCellChanges`，不绕过直接改 grid）——PRD FR-5/FR-8 显式要求"经 store 合法路径提交（commitConversion）"。
- **切 palette 不重匹配**——PRD FR-5 Out of Scope 直接引用"沿用现有'切 palette 不重匹配'的既定行为"，吸收精准。
- **CIEDE2000 性能约定无需 Worker**——PRD FR-7 显式"遵守 project-context 性能约定，无需 Web Worker"，并正确点名 `paletteLabs()` + `findClosestColor` 调用顺序。
- 复用现有 undo/redo 单条历史、复用现有 PNG/PDF 导出——FR-5/FR-8、§6.1 均接住。

### Gap（潜在冲突 / 下游踩坑风险）
1. **【硬约束未点名 — i18n】** project-context 强约束："加任何 i18n key 必须同时补 en + zh，否则类型不过。"本 PRD 将引入**大量新 UI 文案**（库存模式开关、缺色/不足标记、三态可拼性灯、缺口清单人话解读、替代预览、二次确认等）。PRD 未在任何 FR 或 NFR 里提醒"所有新文案必须 en+zh 双语"。这不会让 PRD 逻辑冲突，但下游实现若漏 zh 会**类型门禁直接阻断构建**——值得在 PRD 留一句约束以免反复返工。考虑到 PRD 多处出现中文标记词（"红字""绿灯""还差约 X 颗"），双语义务尤其需要点名。
2. **【硬约束未点名 — PDF 中文】** §6.1 "复用现有 PNG/PDF 导出（无新增）"。但若未来缺口清单/替代信息要进 PDF（FR-11 提到"可导出为文本"），project-context 警告 **PDF 里中文必须经 `addLabel()` 渲染成 canvas，不能用 `doc.text()` 直接写**。PRD 说导出无新增，暂无直接冲突；但"缺口清单导出文本"若被下游理解为进 PDF，存在踩 CJK 乱码坑的风险，建议明确"导出文本 = 纯文本/剪贴板，非 PDF 内嵌"。
3. **【硬约束未点名 — shared rebuild & 导出像素上限】** PRD 大量复用 `shared/color` 引擎（改 shared 必须 rebuild 否则前端静默用陈旧实现）与导出受 `MAX_EXPORT_PX=8000` 约束。这些属实现细节、PRD 不必展开，但 §0 已声明"技术实现属下游架构文档"——可接受。仅作记录：下游架构文档须承接这几条，PRD 未与之冲突。
4. **【无冲突确认】** PRD 的核心机制"换输入即换 palette 子集做匹配"与 project-context 完全兼容（CIEDE2000 引擎换色板输入即可，无需新算法、无需 Worker）。FR-5 要求重匹配"写回 grid 经 commitConversion"与"切 palette 不重匹配"不冲突——因为重匹配是**显式动作**而非切板副作用，PRD 已正确区分这一点。

## 跨文档定性内容核对（情绪 / 调性 / 战略意图是否被 FR 结构丢掉）

- ✅ **用户挫败情绪**（兴冲冲设计完才发现没料 / 肉眼硬凑偏色）——PRD §2.1 情感性 JTBD 接住了，未丢。
- ✅ **"为自己做的工具"调性**——PRD §2.1 + SM-4 接住，是定性内容里最完整的一处。
- ⚠️ **"差异化窗口窄、靠深度续命"的战略紧迫感** —— 被结构化为一个功能点，份量减弱（见简报 gap 2）。
- ❌ **"痛点尚未被证实、只是合理假设"的诚实底色** —— 被 PRD 的确定性叙事完全洗掉（见研究 gap 1）。这是本次核对发现的**最严重的定性丢失**：PRD 读起来像痛点已验证，而两份上游都明确说"未证实"。
- ❌ **"竞品深度待人工复核"的前提脆弱性** —— PRD 当成既成事实（见简报 gap 1 / 研究 gap 2）。

## 建议补丁（供 PM 决定是否回写 PRD）

1. 在 §1 Vision 或 §2.1 增加 `[ASSUMPTION]`：用户痛点（设计完没料、肉眼替代偏色）目前为合理假设，未经 UGC 社区/用户访谈证实（来源：市场研究证据缺口）。
2. 在 §8 Open Questions 增加一条（最高优先）：写实现前/早期，人工到 App Store 复核 BeadMaster/Fuse Beads Studio 等是否已做 CIEDE2000 级感知替代——这是差异化命脉的前置验证。
3. 在 §1 或 §2 引用 Rebrickable 作为"库存驱动范式已被验证有真实刚需"的跨品类背书（同时诚实标注"也说明不新鲜"）。
4. 在某处（NFR 或 §0 结构说明）补一句 i18n 约束：本方向所有新增 UI 文案必须同时提供 en + zh（project-context 强约束，漏 zh 会阻断构建）。
5. 在 §1 或差异化论述里补一句形态对比：竞品均为国内移动 App，pindou 靠 Web 端零安装 + 隐私本地化形成不易复制的形态差异（简报 Vision 卖点）。
